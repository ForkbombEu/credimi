# FCAF missing validators and assertions

Source scan: `../eudi-doc-functional-conformance-assessment/docs/fcaf/suts/wallet_solution/relying_party`.

Public test index used for links: `https://conformance.eudi.dev/latest/fcaf/suts/wallet_solution/relying_party/ws_rp/`.

Concrete tests scanned: 559 `WS_RP_*.md` files.

Important contract correction:

- Validators/assertions should consume raw artifacts from pipeline outputs plus `PipelineEvidenceExtractionActivity` output.
- Pipelines should not be responsible for decoding SD-JWT, mdoc, JWE, DCQL, status lists, or request objects beyond producing raw evidence.
- Pipeline-backed assertions are valid FCAF assertions. If the expected behavior is visible on the phone, the Maestro flow should include `assertVisible` or equivalent checks, and the pipeline result becomes the assertion result.
- Example: `oid4vp.request_accepts_positive_variant` should not be one monolithic Go validator. It should usually be implemented as a pipeline assertion: the pipeline sends the specific request variant, Maestro/verifier checks expected behavior, and FCAF records the pipeline pass/fail. Optional Go validators can inspect the raw request/response evidence for additional protocol-level checks.
